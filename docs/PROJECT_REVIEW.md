## 一、项目概述

cc_bot 是一个基于飞书消息的 Claude Code 移动端操控工具。用户在飞书手机端发消息，bot 在服务器上拉起 Claude CLI 执行，结果通过飞书卡片流式回传。定位是 remote_cli 的轻量移动端补充。

**核心数据**：

| 指标 | 数值 |
|------|------|
| 总代码行数 | ~1,500 |
| Python 模块数 | 7 |
| 外部依赖 | httpx（仅此一个） |
| Git commits | 6 |
| 开发周期 | 约 5 小时（跨 3 天迭代） |
| 稳定运行 | 已验证 |

## 二、架构评估

### 2.1 整体架构

```
飞书 API (轮询)
    ↓
main.py (消息路由 + 状态机)
    ├── feishu_client.py  (REST API)
    ├── claude_runner.py  (Claude 子进程管理)
    ├── stream_handler.py (输出 → 卡片流式)
    ├── conversations.py  (多对话 CRUD)
    ├── history_store.py  (历史持久化)
    ├── state.py          (全局状态)
    └── config.py         (配置)
```

架构风格：**单体无框架**。无 FastAPI、无数据库、无消息队列。所有状态存 JSON 文件，所有通信走 asyncio 协程。

### 2.2 与 remote_cli 的架构对比

| 维度 | remote_cli | cc_bot |
|------|-----------|--------|
| 代码量 | ~31,000 行 | ~1,500 行 |
| 框架 | FastAPI + WebSocket | 纯 asyncio |
| 前端 | Vanilla JS SPA (~14,000 行) | 无（飞书即 UI） |
| 交互方式 | WebSocket 实时推送 | REST 轮询 + 卡片更新 |
| 多用户 | 支持（多 Worker 调度） | 单用户 |
| 执行环境 | 本地 / Docker / SSH | 仅本地 |
| 数据存储 | JSON 文件（20+ store） | JSON 文件（3 个 store） |
| Claude 接入 | stream-json 协议 | stream-json 协议（同款） |

**关键结论**：cc_bot 的核心流式处理逻辑（双协程 stdin 管理、JSONL 解析）直接参照了 remote_cli 的 `claude_process.py`，不是从零造轮子。这是一个正确的工程决策——复用已验证的设计，把精力集中在差异化需求上。

## 三、模块评审

### 3.1 `claude_runner.py` - Claude 子进程管理 ★★★★☆

**亮点**：
- 双协程架构（`_stdin_writer` + `_stdin_close_checker`）参照 remote_cli，经过一次重大重构后稳定
- stream-json 协议解析完整（assistant/user/result/system/error）
- 断网超时检测（600s）+ stderr 网络错误关键词匹配
- 子进程创建失败自动重试 3 次

**不足**：
- `result_event.wait()` 在 `_stdin_close_checker` 中的超时（600s）是硬编码，未放配置
- 没有 Claude 进程内存/CPU 监控
- 单轮模式代码路径与多轮模式耦合在同一函数中

### 3.2 `stream_handler.py` - 卡片流式输出 ★★★☆☆

**亮点**：
- 分级思考计时器（<60s / 60-120s / >120s 不同提示）
- 工具调用自动拆新卡片，回复文本锁定到卡片
- 独立的 `history_buf` 与显示 `buf` 分离——这个设计决策避免了历史记录丢失

**不足**：
- 4000 字符截断后用户看不到完整输出，没有补救机制（如保存到文件）
- 卡片更新失败重试只做 1 次，没有指数退避
- `_throttled_update` 中的节流逻辑和卡片创建逻辑混在一起，职责不够单一

### 3.3 `feishu_client.py` - 飞书 API 封装 ★★★★☆

**亮点**：
- token 管理使用 `asyncio.Lock` + 双重检查，防止并发刷新（这是踩过坑后修的）
- 支持 HTTP 401 和 JSON code 1663 两种 token 过期场景
- 卡片构建干净，schema 2.0 markdown 格式

**不足**：
- `get_messages()` 返回类型 `Optional[list]` 的区分（None=错误, []=无消息）是后来加的，调用方需要理解这个约定
- 没有请求重试/超时配置（依赖 httpx 默认 30s）

### 3.4 `main.py` - 入口 + 消息路由 ★★★☆☆

**亮点**：
- 状态机设计清晰（idle/active）
- 命令路由用字符串前缀匹配，简洁有效
- 多轮消息队列（asyncio.Queue）方案解决了进程保持问题

**不足**：
- 所有逻辑堆在一个文件（~650 行），类方法 20+，职责过多
- `_handle_message` 是上帝方法——包含去重、命令路由、Claude 启动、消息排队
- 没有抽象出 "消息处理器" 或 "命令注册表"，加新命令要改多处
- idle 模式下 `/status` 和 `/help` 可用但 `/list` 等不可用——行为不一致，缺乏设计文档说明

### 3.5 `conversations.py` / `history_store.py` / `state.py` — 持久化 ★★★★☆

**亮点**：
- JSONL 格式选型合理：追加写、按行读、损坏行可跳过
- 原子写入（tmp + rename）防止写一半崩溃
- 从 Claude 原生 session 文件导入历史的 fallback 机制实用

**不足**：
- 三个独立的 store 单例，但没有统一的 store 管理/生命周期
- 历史文件无大小限制，理论上会无限增长
- 历史导入是全局扫描 `~/.claude/projects/*/`，会话多了会慢

### 3.6 `config.py` — 配置管理 ★★★★☆

**亮点**：
- 不依赖 python-dotenv，手动解析 .env，零外部依赖
- 环境变量优先于 .env 文件（12-factor 原则）
- 所有可调参数集中在 dataclass

**不足**：
- 硬编码值散落在其他模块（轮询 30s、超时 600s、自动休眠 1800s），未收归 config

## 五、代码质量

### 5.1 优点

1. **零依赖哲学**：仅依赖 `httpx`，部署只需 Python 3.11+
2. **可读性好**：中文注释 + 类型注解，函数职责基本单一
3. **踩坑记录在代码里**：如 `# 1663 → token 过期但 HTTP 200`、`# 双重检查：拿到锁后再次确认`
4. **Git 记录干净**：6 个 commit，每个有清晰的变更说明
5. **有参照而非盲抄**：remote_cli 的双协程架构是被理解后复用的

### 5.2 问题

1. **无测试**：零单元测试、零集成测试。改代码靠手工验证
2. **上帝类**：`Bot` 类承载了太多职责（路由、进程管理、状态机、队列管理）
3. **print() 日志**：无日志级别、无文件轮转、排查问题靠 grep
4. **硬编码散落**：超时时间、轮询间隔、截断长度分散在各模块
5. **无错误恢复策略**：bot 崩溃后需手动重启，没有 health check 或 watchdog

## 七、总结

### 项目定位

cc_bot 是一个**有效的移动端 Claude 操控工具**。它不是 remote_cli 的替代品，而是互补品——remote_cli 用于工位深度开发，cc_bot 用于移动端快速交互。

### 核心价值

- **真实的用户痛点**：手机上看不了 Claude 输出 → 解决
- **真实的开发效率**：5 小时从零到可用，6 轮迭代稳定
- **真实的工程决策**：参照而非重写、踩坑后修复、有 git 可回退

### 一句话评价

> 一个代码量不大但设计用心的移动端开发助手，核心流式引擎设计正确，功能集刚好够用，技术债务可控。定位为个人工具完全合格，如要扩展为团队工具需要重构。

## 八、v1.1 优化后评估（2026-06-22）

### 优化内容

基于 v1 评审报告的 P1-P3 技术债清单，完成 3 轮 7 项优化：

| Round | 改动 | 效果 |
|:---:|------|------|
| 1.1 | `print()` → `logging` | 日志分级、格式统一、抑制 httpx 噪音 |
| 1.2 | 命令路由 if/elif → dict 表 | `_handle_message` 从 70 行减至 2 行，加命令只需在 EXACT/PREFIX dict 中加一行 |
