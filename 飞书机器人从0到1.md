## 1. 创建飞书应用

### 1.1 进入飞书开放平台

打开 [https://open.feishu.cn](https://open.feishu.cn)，用飞书账号登录。

### 1.2 创建企业自建应用

1. 点击「创建应用」→「企业自建应用」
2. 应用名称：`Claude Assistant`（可自定义）
3. 应用描述：随意填写
4. 创建完成后进入应用详情页

### 1.3 添加机器人能力

左侧菜单 →「添加应用能力」→ 找到「机器人」→ 添加

## 2. 配置权限

左侧菜单 →「权限管理」→ 搜索并开通以下权限：

| 权限代码 | 用途 |
|---------|------|
| `im:message:send_as_bot` | 以机器人身份发消息 |
| `im:message` | 接收用户消息 |
| `im:chat:readonly` | 读取聊天信息 |
| `contact:contact.base:readonly` | 查询用户信息 |

> **方法一**：逐个搜索权限 → 勾选 → 批量开通
>
> **方法二**：点「批量导入」，粘贴 JSON：
> ```json
> {
>   "scopes": {
>     "tenant": [
>       "im:message:send_as_bot",
>       "im:message",
>       "im:chat:readonly",
>       "contact:contact.base:readonly"
>     ]
>   }
> }
> ```

每个权限开通后立即生效（企业自建应用无需审核）。

## 4. 获取用户 Open ID

机器人发消息需要一个 `open_id` 来标识接收者。

### 方法：API 查询（推荐）

开通 `contact:contact.base:readonly` 权限后，用手机号查：

```bash
TOKEN=$(curl -s -X POST \
  'https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal' \
  -H 'Content-Type: application/json' \
  -d '{"app_id":"cli_xxx","app_secret":"xxx"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['tenant_access_token'])")

curl -s "https://open.feishu.cn/open-apis/contact/v3/users/batch_get?user_ids=你的手机号" \
  -H "Authorization: Bearer $TOKEN"
```

返回结果中的 `open_id`（以 `ou_` 开头）就是用户的 ID。

或者直接查用户列表：

```bash
curl -s 'https://open.feishu.cn/open-apis/contact/v3/users?page_size=10' \
  -H "Authorization: Bearer $TOKEN"
```

### 5.4 feishu_listener.py（核心文件）

完整代码见项目中的 `feishu_listener.py`。核心设计：

```
┌──────────────┐    每2秒轮询     ┌──────────────┐
│  飞书服务器   │ ←────────────── │  Bot 监听器   │
│              │  GET /messages  │              │
│  用户发消息   │ ──────────────→ │  检查最新消息  │
│              │  返回消息列表    │              │
│              │                │  msg_id 不同?  │
│              │                │      ↓        │
│              │                │  claude -p     │
│              │                │  --continue    │
│              │                │      ↓        │
│              │  POST 卡片消息  │  发送结果      │
│              │ ←────────────── │              │
└──────────────┘                └──────────────┘
```

**关键设计决策**：

| 决策 | 原因 |
|------|------|
| REST 轮询（不用 WebSocket） | WS 事件投递不稳定，SDK 线程模型复杂 |
| 每条消息只发一条回复 | 避免消息轰炸 |
| 在任何 `await` 之前设置去重标记 | asyncio 协程在 await 处切换，必须先标记 |
| `--continue` 接续会话 | 保持上下文记忆 |
| `/new` 命令清空上下文 | 删除 session JSONL 文件 |

### 5.5 feishu_notify.py（工具库）

可在任意 Python 脚本中导入使用：

```python
from feishu_notify import FeishuBot
import asyncio

async def main():
    bot = FeishuBot()
    
    # 发送文本
    await bot.send_text("ou_xxx", "训练完成")
    
    # 发送任务结果卡片（带颜色）
    await bot.send_task_result(
        receive_id="ou_xxx",
        task_name="GRPO 训练",
        status="success",  # success/failed/running/cancelled
        result_summary="cut=42.1, imbalance=0.008"
    )

asyncio.run(main())
```

### 5.6 启动/停止脚本

**feishu_start.sh**：
```bash
#!/bin/bash
cd "$(dirname "$0")"
pkill -9 -f "feishu_listener.py" 2>/dev/null
sleep 1
rm -f .feishu_listener.pid
nohup python3 feishu_listener.py > .feishu_listener.log 2>&1 &
echo "Bot 已启动 (PID: $!)"
```

**feishu_stop.sh**：
```bash
#!/bin/bash
cd "$(dirname "$0")"
if [ -f .feishu_listener.pid ]; then
    PID=$(cat .feishu_listener.pid)
    kill "$PID" 2>/dev/null && echo "已停止 (PID: $PID)"
    rm -f .feishu_listener.pid
fi
```

## 7. 踩坑记录

### 坑 1：不要用 WebSocket 事件订阅

**现象**：SDK WebSocket 连接成功、有心跳，但收不到任何消息事件。

**根因**：飞书事件订阅推送方式需设为"长连接"，且 `im.message.receive_v1` 事件需正确订阅。即使配置正确，同一 app_id 的多个 WS 连接会随机路由消息。

**解决**：改用 REST API 轮询（`GET /im/v1/messages`），简单可靠，3 秒延迟完全可接受。

### 坑 2：同一条消息被重复处理多次

**现象**：发一条消息，Bot 回复 3-4 次。

**根因 A**：多个 Bot 进程同时运行。`./feishu_start.sh` 如果没有 `pkill -9 -f` 清理旧进程，重启后会累积多个实例。

**解决**：启动脚本用 `pkill -9 -f` 强制杀掉所有旧进程。

**根因 B**：asyncio 去重标记在 `await` 之后才设置。

```python
# ❌ 错误：await 之间协程切换，去重失效
async with self._lock:  # await!
    self._last_processed_id = msg_id  # 太晚了

# ✅ 正确：在任何 await 之前设置
self._last_processed_id = msg_id  # 先标记
async with self._lock:  # 再等锁
```

**解决**：去重标记必须在第一个 `await` 之前赋值。

### 坑 3：更新卡片消息（PATCH）不可靠

**现象**：先发"正在思考..."，再 PATCH 更新为结果。有时更新失败，fallback 发新消息，用户看到两条。

**解决**：不自找麻烦。直接发一条完整结果消息，不做更新。

### 坑 4：Python 3.8 类型注解兼容

**现象**：`TypeError: 'type' object is not subscriptable`

**解决**：文件顶部加 `from __future__ import annotations`

### 坑 5：`claude` 子进程继承 `CLAUDECODE` 环境变量

**现象**：`Error: Claude Code cannot be launched inside another Claude Code session`

**解决**：创建子进程时 `env.pop("CLAUDECODE", None)`

### 坑 6：`docker exec` 缺 `-i` 导致 stdin 透传失败

**现象**：Docker 容器内 claude 收不到输入，exit code 0 无输出。

**解决**：`docker exec` 必须加 `-i`（Keep STDIN open）。

### 坑 7：飞书 @mention 格式不统一

**现象**：HTTP 回调和 WS 长连接中 @mention 格式不同。

**解决**：两种格式都要处理：
```python
text = re.sub(r"<at[^>]*>.*?</at>", "", text)   # HTTP 回调
text = re.sub(r"@_user_\d+\s*", "", text)        # WS 长连接
```

### 坑 8：PATCH vs PUT 的区别

**现象**：用 PATCH 更新文本消息失败（400）。

**根因**：lark-oapi 中 `PatchMessageRequest`（PATCH）仅支持卡片消息；`UpdateMessageRequest`（PUT）支持文本消息；`ReplyMessageRequest` 用于回复原消息。

**解决**：用 POST 发送新卡片消息，不更新已有消息。
